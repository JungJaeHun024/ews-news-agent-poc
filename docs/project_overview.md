\# Project Overview



\## 프로젝트명



건설산업 EWS 비정형 언론보도 수집·검증 AI Agent PoC



\## 프로젝트 목적



본 프로젝트는 건설산업 조기경보시스템(EWS)의 비정형 데이터 항목인 건설경기 관련 언론보도를 대상으로, 특정 기준일자·언론사·키워드 조건에 따라 후보 기사를 수집하고, 원문 접근성·메타데이터 확보 가능성·본문 추출 가능성·RAG 활용 가능성을 검증하는 것을 목적으로 한다.



사용자는 다음과 같은 자연어 질의를 입력할 수 있다.



> 오늘일자 기준으로 조중동의 건설경기 관련 뉴스 기사를 탐색해줘



Agent는 해당 질의를 기준일자, 대상 언론사, 검색 키워드로 구조화한 뒤 Google News RSS 기반 후보 기사 수집부터 원문 URL 해석, 기사 검증, NVIDIA NIM 기반 관련성 판단, SQLite/Chroma 저장, 자연어 응답 출력까지 수행한다.



\## 문제 정의



건설산업 조기경보시스템에서 언론보도 데이터는 다음과 같은 이유로 단순 수집만으로는 활용하기 어렵다.



\- 뉴스 데이터는 비정형 데이터이다.

\- Google News RSS URL은 실제 언론사 원문 URL이 아니라 중간 URL인 경우가 많다.

\- 언론사별 HTML 구조가 달라 작성자, 작성일자, 본문 추출 품질이 일정하지 않다.

\- 검색어 기반 후보 수집 결과에는 관련 없는 기사도 포함될 수 있다.

\- RAG에 모든 후보 기사를 저장하면 검색 품질이 낮아질 수 있다.

\- 기사 전문 저장은 저작권 및 이용권한 검토가 필요하다.



따라서 본 프로젝트는 LLM이 직접 뉴스를 임의로 탐색하는 방식이 아니라, 코드 기반 수집·검증 파이프라인과 제한적 LLM 판단 Agent를 결합한 구조로 설계하였다.



\## 핵심 접근 방식



본 프로젝트의 핵심 설계 원칙은 다음과 같다.



1\. 수집과 검증은 코드 기반으로 수행한다.

2\. LLM은 검증된 후보 기사에 대해서만 관련성과 위험 이벤트를 판단한다.

3\. 전체 기사 전문은 Vector DB에 저장하지 않는다.

4\. 검증된 메타데이터, 요약, 위험 이벤트, 판단 근거, 원문 URL 중심으로 저장한다.

5\. LLM API rate limit을 고려하여 최대 호출 수와 호출 간격을 제한한다.

6\. LLM 호출 실패 시 규칙 기반 fallback을 사용한다.



\## 주요 구현 기능



\### 1. 자연어 질의 기반 수집 조건 생성



사용자 자연어 질의를 기준일자, 언론사, 검색 키워드로 변환한다.



예시:



\- 오늘 → 현재 날짜

\- 조중동 → 조선일보, 중앙일보, 동아일보

\- 건설경기 관련 → 건설경기, 건설수주, 미분양, PF 부실, 공사비 상승, 건설투자, 주택시장



\### 2. Google News RSS 후보 기사 수집



구조화된 검색 조건을 바탕으로 Google News RSS 검색 URL을 생성하고 후보 기사 URL을 수집한다.



\### 3. Google News URL Resolver



Google News RSS에서 제공하는 중간 URL을 실제 언론사 원문 URL로 변환한다.



\### 4. 원문 URL 접근성 검증



원문 URL에 접근하여 HTTP status, final URL, final domain, content type, 접근 결과를 확인한다.



\### 5. 기사 메타데이터 및 본문 검증



기사 제목, 작성자, 작성일자, 설명, canonical URL, 본문 존재 여부, 본문 길이, 본문 일부 preview를 추출한다.



\### 6. 규칙 기반 위험 이벤트 태깅



건설경기 관련 위험 이벤트를 규칙 기반으로 태깅한다.



주요 이벤트 유형은 다음과 같다.



\- 건설수주 감소

\- 미분양 증가

\- PF 부실

\- 공사비 상승

\- 착공 지연

\- 주택시장 위험

\- 건설투자 부진

\- 기업 재무 건전성



\### 7. NVIDIA NIM 기반 관련성 판단



NVIDIA NIM API를 활용하여 기사 관련성, 주요 위험 이벤트, 요약, 판단 근거를 생성한다.



단, 모든 후보 기사에 LLM을 호출하지 않고, 검증 조건과 위험 신호를 만족하는 기사에 대해서만 제한적으로 호출한다.



\### 8. SQLite 저장



각 단계의 결과를 SQLite 테이블로 저장한다.



주요 테이블은 다음과 같다.



\- search\_tasks

\- article\_candidates

\- article\_access\_checks

\- article\_metadata

\- rule\_risk\_events

\- llm\_judgements

\- cross\_source\_checks



\### 9. Chroma Vector DB 저장



검증 기준을 통과한 기사만 Chroma Vector DB에 저장한다.



저장 기준은 다음과 같다.



\- 원문 URL 접근 가능

\- 언론사 도메인 일치

\- 실제 기사 여부 확인

\- 본문 존재

\- 관련성 판단 통과

\- Google News 중간 URL 제외



\### 10. Excel 검토표 생성



수집가능성 검토표에 활용할 수 있는 항목별 판단 결과와 근거를 생성한다.



주요 검토 항목은 다음과 같다.



\- 원문 확보 가능성

\- 출처 URL 확보 가능성

\- 메타데이터 확보 가능성

\- 자동수집 가능성

\- 수집 안정성

\- 저작권·이용권한 검토

\- RAG 활용 가능성

\- 위험 이벤트 추출 가능성

\- 교차확인 가능성



\## 프로젝트 성격



이 프로젝트는 운영용 뉴스 크롤링 시스템이 아니라 실무형 PoC이다.



목적은 특정 언론보도 데이터를 무제한 수집하는 것이 아니라, EWS 데이터 구축 관점에서 비정형 뉴스 데이터가 AI-Ready 데이터로 전환될 수 있는지 검증하는 것이다.



\## 기대 효과



\- 비정형 뉴스 데이터의 수집가능성 검토 자동화

\- 건설경기 관련 위험 이벤트 후보 탐지

\- RAG 적재 전 데이터 품질 검증

\- LLM 기반 판단과 규칙 기반 검증의 결합

\- 데이터 기획, 데이터 엔지니어링, AI Agent 개발 관점의 실무형 PoC 산출
